# 电商购物车与支付自动化测试

## 快速开始

接到任务后按顺序执行：

1. **确认测试层级**：API / UI(E2E) / 混合；若无说明，优先 API 覆盖业务逻辑，E2E 覆盖关键用户路径
2. **读取项目约定**：检查现有测试框架（pytest、Playwright、Selenium 等）、目录结构、fixture、Page Object
3. **对齐被测系统**：识别购物车 ID 策略（userId / sessionId / token）、库存扣减时机、支付回调方式
4. **输出测试资产**：测试用例清单 → 自动化代码 → 必要的数据工厂与 mock

## 测试范围矩阵

| 模块 | 必测场景 | 常见边界 |
|------|----------|----------|
| 购物车 | 加购、改数量、删商品、清空、合并游客车 | 库存不足、下架、限购、失效 SKU、并发改数量 |
| 结算 | 地址/运费/优惠券/积分/税费计算 | 券过期、门槛不满足、地址不可达、价格变动 |
| 支付 | 发起支付、成功、失败、超时、重复支付 | 幂等、回调乱序、部分退款、沙箱与生产隔离 |
| 订单 | 状态流转、取消、超时关单 | 支付中取消、已支付取消规则 |

## 用例设计规范

### 命名

```
test_<模块>_<场景>_<预期结果>
```

示例：`test_cart_add_item_increases_quantity_when_same_sku`

### 结构（AAA）

- **Arrange**：用户/商品/库存/优惠券等前置数据
- **Act**：调用 API 或 UI 操作
- **Assert**：响应码、业务码、DB/缓存、UI 文案、订单状态

### 优先级

- **P0**：主路径（加购 → 结算 → 支付成功 → 订单完成）
- **P1**：常见异常（库存不足、支付失败、券不可用）
- **P2**：边界与并发（限购、幂等、回调重试）

### 每条用例必须包含

| 字段 | 说明 |
|------|------|
| ID | 如 `CART-001` |
| 标题 | 一句话描述 |
| 前置条件 | 登录态、商品状态、库存 |
| 步骤 | 可执行的操作序列 |
| 预期结果 | 可断言的观测点 |
| 测试数据 | SKU、数量、金额、券码 |
| 清理 | 是否回滚订单/释放库存 |

完整模板见 [test-case-template.md](test-case-template.md)。

## 购物车测试清单

生成用例时至少覆盖：

```
- [ ] 空购物车展示
- [ ] 单 SKU 加购（数量=1）
- [ ] 同 SKU 重复加购（合并数量）
- [ ] 修改数量（合法 / 超库存 / 低于最小起购）
- [ ] 删除单个商品
- [ ] 批量删除 / 清空购物车
- [ ] 下架或失效 SKU 的处理（不可结算提示）
- [ ] 登录前后游客购物车合并（以业务规则为准）
- [ ] 购物车金额汇总（单价×数量、小计、不含运费）
- [ ] 并发加购同一 SKU（最终数量与库存一致）
```

## 结算测试清单

```
- [ ] 默认地址与运费计算
- [ ] 切换地址导致运费变化
- [ ] 可用优惠券抵扣（满减 / 折扣 / 包邮）
- [ ] 不可用优惠券（过期、未达门槛、品类限制）
- [ ] 积分抵扣上限
- [ ] 结算页价格与购物车一致
- [ ] 提交订单后库存预占或扣减（按系统设计断言）
- [ ] 提交订单生成唯一 orderNo
```

## 支付测试清单

```
- [ ] 创建支付单（金额、币种、orderNo 绑定）
- [ ] 支付成功回调 → 订单状态变为已支付
- [ ] 支付失败 → 订单保持待支付，可重试
- [ ] 重复回调幂等（同一 paymentId 不重复记账）
- [ ] 支付超时 / 用户取消
- [ ] 0 元单 / 纯积分单（若支持）
- [ ] 沙箱密钥与生产隔离（测试环境禁止真实扣款）
```

## 自动化实现原则

### 分层建议

| 层级 | 职责 | 工具示例 |
|------|------|----------|
| API | 购物车/订单/支付接口契约 | pytest + requests/httpx |
| UI | 关键路径冒烟 | Playwright / Selenium |
| 契约 | 支付网关 mock | WireMock、responses、本地 stub |

### 数据与隔离

- 每个用例独立 test user 或独立 cart session，避免串扰
- 商品与库存用 factory 或 fixture 创建，测试后清理或标记废弃
- 支付使用**沙箱/mock**，禁止在自动化中调用生产支付

### 等待与断言

- 异步回调：轮询订单状态 + 超时，而非固定 sleep
- 金额断言用 `Decimal` 或分（cents）整数，避免浮点误差
- 断言顺序：HTTP 状态 → 业务 code → 领域对象（order.status）

### 项目适配

编写代码前必须：

1. 搜索现有 `test_*.py`、`conftest.py`、`*Page*.py`
2. 复用已有 client、auth fixture、base URL 配置
3. 匹配项目命名与目录（如 `tests/api/`、`tests/e2e/`）

## 输出格式

### 1. 用例清单（Markdown 表格）

```markdown
| ID | 优先级 | 模块 | 标题 | 类型 |
|----|--------|------|------|------|
| CART-001 | P0 | 购物车 | 加购后数量正确 | API |
```

### 2. 自动化代码

- 一个场景一个 test 函数，避免巨型测试
- 提取重复步骤为 helper/fixture，不复制粘贴整段流程
- 失败时附加：request/response 摘要、orderNo、trace id

### 3. 变更说明

简要列出新增文件、依赖的环境变量（如 `PAYMENT_SANDBOX_KEY`）、如何本地运行（仅说明命令，**不要自动执行测试**，除非用户明确要求）。

## 决策树

**仅要测试用例文档？** → 输出清单 + 详细步骤，不写代码

**要可运行自动化？** → 先 API 用例，再补 1～2 条 E2E 冒烟

**支付无法连真实网关？** → 使用 mock 回调接口 + 断言订单状态机

**库存/订单偶发失败？** → 检查并发与清理；增加 retry 仅用于环境 flaky，不掩盖业务 bug

## 反模式（避免）

- 在 UI 测试中验证复杂金额计算（应在 API 层断言）
- 多条业务断言堆在一个 test 里导致难以定位失败
- 硬编码 sleep 等待支付回调
- 共用同一订单号跑并行 CI
- 测试环境使用真实银行卡或生产支付密钥
